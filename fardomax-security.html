SECURITY ARCHITECTURE

Security White Paper

A complete technical breakdown of how Fardomax protects your files from end to end — the cryptography, the protocols, and the threat model.

01
🔑
Encryption Core — AES-256-GCM

Every byte of every file is encrypted with AES-256-GCM before it leaves your browser. This is a symmetric authenticated cipher — widely regarded as the gold standard for data encryption.

Algorithm
AES-GCM
Galois/Counter Mode — provides confidentiality + authenticated integrity in one pass
Key Length
256-bit
2²⁵⁶ possible keys — brute force would take longer than the age of the universe
IV Size
96-bit random
Fresh random IV per chunk using crypto.getRandomValues — prevents IV reuse
Auth Tag
128-bit GCM Tag
Decryption fails on any bit-flip — tamper-evident by design
Key Source
Web Crypto API
crypto.subtle.generateKey — OS-level CSPRNG, never predictable
Chunk Size
64 KB
Each chunk independently encrypted with its own IV — streaming-friendly and secure

GCM mode provides Authenticated Encryption with Associated Data (AEAD). This means both confidentiality (nobody can read the data) and integrity (nobody can tamper with it without detection) are guaranteed by a single cryptographic operation.

⚠ Critical Security Note

The share URL contains your encryption key. Share it only through trusted channels. Anyone with the URL can decrypt the files. Use end-to-end encrypted messaging (Signal, WhatsApp, iMessage) to share the URL — not SMS or email where possible.

URL fragments (the #part) are a browser security boundary — they are never included in HTTP requests. This is defined in RFC 3986 and enforced by every major browser. Even the PeerJS signal server that receives the rest of the URL never receives the key.

03
🛡️
Transport Layer Security

Beyond application-layer AES-256, all network communication uses multiple layers of transport encryption:

  • WebRTC Data Channel (DTLS 1.2+) — All WebRTC data channels are mandated by the spec (RFC 8261) to use DTLS. This means even before our AES-256 layer, the data is protected by transport-level encryption.
  • Signal Server (TLS 1.3) — The PeerJS signaling connection uses TLS 1.3, ensuring signaling metadata (peer IDs, SDP, ICE candidates) is encrypted in transit.
  • TURN Relay (TLS/TCP 443) — When using TURN relay, we prefer port 443/TCP which routes through TLS, bypassing firewalls and adding a second transport encryption layer on top of our AES-256.
  • Double Encryption on TURN — If a TURN relay is used, your data is protected by both AES-256-GCM (your key, application layer) AND DTLS (transport layer). The TURN server operator sees only opaque double-encrypted ciphertext.

🌐
WebRTC Security Properties
  • Mandatory DTLS — The WebRTC specification (W3C + IETF) requires all data channels to use DTLS encryption. Browsers cannot disable this. It is enforced at the protocol level.
  • Certificate Fingerprinting — WebRTC uses self-signed certificates with fingerprints exchanged via the signaling channel (SDP). This prevents MITM attacks even if the signal server is compromised, as long as the SDP exchange itself is over TLS.
  • ICE Credential Security — Each ICE session uses randomly generated username/password credentials, preventing session hijacking.
  • No Plugin Required — Runs in the browser's native WebRTC stack — no plugins, extensions, or software installs that could be backdoored.
  • Same-Origin Policy — The Web Crypto API operates under browser sandbox restrictions. Keys generated in one origin cannot be accessed by scripts from another origin.

Security Best Practices for Users
✓ DO
  • Share the link via Signal, WhatsApp, or iMessage
  • Use HTTPS when hosting this page
  • Verify the recipient received files before closing tab
  • Generate a new link for each transfer session
  • Use Chrome, Firefox, or Edge for best WebRTC support
  • Keep your browser updated for latest security patches
  • Transfer over trusted networks when possible
✗ DON'T
  • Share the link via unencrypted SMS
  • Post the share link publicly (social media, forums)
  • Reuse the same share link for multiple sessions
  • Leave sender tab open indefinitely after transfer
  • Transfer highly sensitive data on compromised devices
  • Use on outdated browsers without WebCrypto support
  • Share link via email on unsecured mail servers
