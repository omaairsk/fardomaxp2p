TECHNICAL GUIDE

How It Works

The complete technical picture — from clicking "Generate Link" to a downloaded file. No magic, no black boxes, just WebRTC and cryptography working together.

QUICK START — 3 STEPS

Deep Dive — Step by Step
1
🔑Key Generation
When you click "Generate Secure Link", your browser calls crypto.subtle.generateKey({name:'AES-GCM', length:256}, true, ['encrypt','decrypt']). This generates a cryptographically random 256-bit symmetric key using your operating system's entropy source. The key is then exported to raw bytes and hex-encoded into a 64-character string.
Implementation: Web Crypto API · OS CSPRNG · Never predictable
2
🆔Peer ID Registration
A unique peer ID (e.g., fmx-lx4k9abc123) is generated from a timestamp + random string, then registered with the PeerJS signal server over a TLS 1.3 WebSocket connection. The signal server now knows this ID exists and can relay connection offers to it. No files or keys are sent to the signal server.
Signal server sees: Peer ID only · No file data · No encryption keys
3
🔗Share URL Construction
The share URL is assembled as: https://[domain]/[page].html#[peerID]|[keyHex]. The part after # (the URL fragment) is never transmitted to any server — this is a fundamental RFC 3986 browser security property. The key lives only on your device and travels to the recipient only via the URL you share.
URL fragment: Never sent to servers · Browser security boundary · RFC 3986
4
🤝WebRTC Handshake
The receiver pastes the link. Their browser extracts the peer ID from the URL fragment and sends a connection offer to the sender via the signal server (an SDP offer). The sender's browser responds with an SDP answer. Both sides then exchange ICE candidates — network address/port pairs to attempt direct connection. This handshake is the "signaling" phase.
Handshake: SDP offer/answer · ICE candidates · Via signal server (TLS) · No file data
5
🌐ICE / TURN Connection
Both browsers use the ICE (Interactive Connectivity Establishment) framework to find the best route. Direct P2P: If both peers can reach each other (e.g., same network), a direct UDP connection is established. TURN relay: If NAT/firewall prevents direct connection, an encrypted relay via our TURN servers is used. All relay traffic is double-encrypted (AES-256-GCM + DTLS).
Connection: Direct UDP preferred · TURN relay fallback · All traffic encrypted
6
🔒Encrypted Chunked Transfer
The sender reads the file(s) in 64 KB chunks. Each chunk is individually encrypted with AES-256-GCM using a fresh random 96-bit IV: encryptChunk(chunkBuffer). The encrypted chunk (IV + ciphertext + GCM auth tag) is sent over the WebRTC data channel. The receiver decrypts each chunk as it arrives, then assembles them into the complete file in memory.
Transfer: 64KB chunks · Unique IV per chunk · Stream-safe · Backpressure managed
7
💾Download
Once all chunks of a file are received and decrypted, the browser assembles a Blob from the decrypted chunk array and creates an ObjectURL pointing to it. A download button appears with the original filename. When you click it, the browser saves the file directly from memory — it never touched a server.
Download: Direct from browser memory · Original filename preserved · Blob ObjectURL

Frequently Asked Questions
Does Fardomax work across different networks (home → office)?+

Yes. This is the primary use case. When two peers are on different networks, WebRTC's ICE framework tries to establish a direct connection. If that fails (strict NAT or firewall), the TURN relay servers transparently take over. The transfer continues seamlessly — you may notice it's slightly slower on a relay, but it will work anywhere in the world.

Is there really no file size limit?+

Technically, yes — there is no software-imposed limit. The file is read in 64 KB chunks and streamed through the peer connection. Practical limits depend on: your network speed (larger files take longer), browser RAM (for very large files, buffering in memory may be a constraint on low-RAM devices), and whether the sender keeps their tab open for the full duration. Files of 10–100+ GB have been successfully transferred over local P2P connections.

What happens if the connection drops mid-transfer?+

Currently, Fardomax does not support resume-on-reconnect. If the connection drops, the receiver will need to click Retry and the sender will need to keep their page open. The transfer restarts from the beginning. This is a known limitation — for very large files on unstable connections, we recommend a stable WiFi network. Resumable transfer is a planned feature.

Can I send to multiple recipients at once?+

The current version supports one sender-to-one-receiver connection per link. To send to multiple recipients, generate a new link and share it with each person separately. Each connection creates a fresh encryption key and peer session. Note: each recipient must connect while the sender's page is still open.

How long does the share link stay valid?+

The link is valid only while the sender's browser tab remains open. Once the sender closes or refreshes their tab, the peer ID is deregistered and the link becomes invalid. There is no server-side expiry timer — validity is purely tied to the sender's active connection. This is a deliberate security feature: links cannot be used after the sender is done.

What browsers are supported?+

Best support: Chrome 90+, Edge 90+, Firefox 85+, Opera 76+. Limited support: Safari 14.1+ (WebRTC data channels work but may be slower). Not supported: Internet Explorer (no WebRTC). The requirements are: WebRTC Data Channels, Web Crypto API (crypto.subtle), and ES6+. All modern browsers meet these requirements. Mobile browsers (Chrome for Android, Firefox for Android, Safari for iOS) also work.

Can I use this offline or on a local network?+

For a local network transfer (same WiFi/LAN), you still need internet access for the initial signaling handshake (connecting to PeerJS signal server). Once the P2P connection is established, all data travels locally. Fully offline transfers are not currently supported. If you have your own PeerJS server running locally, you could modify the PCFG config to point to it.
